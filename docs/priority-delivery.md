# 云旅无忧优先级交付清单

## 1.2 首期交付范围

- 用户端：微信小程序、iOS App、Android App，统一体验，覆盖首页、AI 管家、活动地图、紧急求助、智能硬件、旅居管家、订单、消息、我的。
- 人工向导端：小程序/App 内人工向导角色端，覆盖接单大厅、订单详情、服务中、异常上报、我的订单、今日收入、上线接单设置。
- 商户端：小程序/App 内商户角色端，覆盖服务发布、订单预约、确认报价、服务执行、完成上报、评价售后。
- 管理后台：PC Web 管理后台，覆盖用户、老人、人工向导、商户、服务、订单、活动、异常、设备、数据看板、系统配置。
- 硬件数据：小云机器人、智能手环及扩展智能设备，首期以模拟数据/接口预留为主。
- AI 能力：智能管家问答、服务推荐、异常解释，首期以通用大模型、预设知识库和规则推荐实现。

## 1.3 两周 MVP 原则

- 先完成可演示、可试运营、可验收的核心闭环，不追求首期全量商业化。
- 真实硬件、支付结算、医保/医院深度接口、复杂 AI 诊断等高依赖能力首期仅做接口预留或模拟。
- 前端优先采用跨端技术栈，减少微信小程序、iOS、Android 三端重复开发。
- 管理后台优先满足运营、调度、异常处理和数据查看，复杂 BI 分析后续迭代。

## P0 核心闭环必需功能

状态：已完成。否则会影响演示和试运营。

- 用户端：登录、AI 管家、活动地图、旅居管家服务、SOS、订单、消息。
- 向导端：上线接单、查看任务、接单、开始服务、异常上报、完成服务、今日收入。
- 商户端：发布服务、接收预约、确认方案/报价、完成服务、查看评价。
- 管理后台：数据概览、审核向导/商户、管理活动、处理 SOS、派单、查看订单闭环。
- 数据闭环：用户、健康、服务、商户、评价、服务请求数据沉淀和后台查询。
- 工程保障：模拟数据库、鉴权权限、状态机、一键重置演示数据、验收脚本。

## P1 重要增强功能

状态：已完成。时间不足时可简化，但当前已纳入试运营版本。

- 活动报名必须填写姓名、手机号和人数，并写入报名数据。
- 社群、打卡、美食、交通、商城、志愿等入口生成真实服务请求，后台可处理。
- 向导异常上报生成后台异常事件，处理后状态同步。
- 商户服务发布进入待审核，后台审核后上架。
- 后台统一展示 P0/P1/P2 交付状态、服务请求、评价、活动和审核数据。

## P2 后续迭代功能

状态：已预留入口或接口。首期不伪装成真实商业化接入。

- 支付/退款：`/api/integrations/payment/request`
- 短信/电话通知：`/api/integrations/sms/request`
- 智能设备真实协议：`/api/integrations/hardware/request`
- 医保/医院深度接口：`/api/integrations/medical-hospital/request`
- 对象存储/图片上传：`/api/integrations/storage/request`
- 真实大模型服务：`/api/integrations/llm/request`
- 复杂 AI 诊断：`/api/integrations/ai-diagnosis/request`
- iOS/Android/小程序打包链路：`/api/integrations/app-build/request`
